You don't want to repeat *Create Your Agent* → *Give It a Brain* → *Identity & Soul* by hand for every specialist. So the fleet is described in **one spec** and brought up by **one helper** — create, scope the policy, stage the persona — in a loop.

## The spec — one line per agent

`fleet.txt` is the whole fleet in one file — **one agent per line, three whitespace-separated columns**:

```text
# name      backend(host:port, or "-")                              skill(@scope/name, or "-")
logs        loki.monitoring.svc.cluster.local:3100                  -
metrics     kps-kube-prometheus-stack-prometheus.monitoring.svc.cluster.local:9090   -
traces      tempo.monitoring.svc.cluster.local:3200                 -
events      loki.monitoring.svc.cluster.local:3100                  -
analyst     -                                                       -
```

Each column:

- **`name`** — the agent's name. It's also the folder `manifests/openclaw/fleet-roles/<name>/`, where its `IDENTITY.md` and `SOUL.md` live. So `logs` becomes a sandbox named `logs` with the persona from `fleet-roles/logs/`.
- **`backend`** — the **single** `host:port` this agent's egress policy will allow (its one tool). `loki…:3100` for the logs agent, Prometheus `:9090` for metrics, Tempo `:3200` for traces, and `loki…:3100` again for `events` (which reads the Kubernetes event stream an event-exporter ships into Loki). Use **`-`** for *no egress* — that's the `analyst`, which only reasons over what the others found.
- **`skill`** — an optional registry skill to install as a first-class tool (e.g. a `@workshop/promql` skill giving `promql(query)` instead of hand-rolled `curl`). **`-`** means none — the agent still works, querying its backend via the instructions in its `SOUL.md`. Publish a skill with the *Skills & Governance* flow, then drop `@workshop/<name>` in this column.

<Callout type="note" title="Columns are whitespace-separated, not colon-separated">
The backend is itself a <code>host:port</code> (it contains a colon), so the columns are split on <strong>whitespace</strong>, not <code>:</code>. Line them up however you like; <code>fleet.sh</code> reads the first three fields. Lines starting with <code>#</code> are comments.
</Callout>

## The helper — `scripts/fleet.sh`

For each line it creates a sealed sandbox, writes a **policy scoped to just that backend** (everything else stays deny-by-default, plus the in-cluster skill registry), and stages the agent's persona from `manifests/openclaw/fleet-roles/<name>/`.

Bring the fleet up (this creates the sandboxes, scopes each policy, and stages each persona — it prints progress per agent):

```bash
./scripts/fleet.sh up fleet.txt
```

Then watch them reach Ready (`status` wraps `openshell sandbox list`):

```bash
./scripts/fleet.sh status
```

### Look at what's running

See the sandbox **pods** on the cluster (each agent is a sealed pod — they live in the `openshell-sandboxes` namespace; the gateway control plane stays in `openshell`):

```bash
kubectl get pods -n openshell-sandboxes
```

The same fleet as **Sandbox custom resources** — what the OpenShell gateway reconciles into those pods:

```bash
kubectl get sandboxes -n openshell-sandboxes
```

And the **gateway's own view** of the fleet (its phase) — `openshell` talks to the gateway, not the Kubernetes API:

```bash
openshell sandbox list
```

Inspect any one agent's **effective policy** — proof its egress is scoped to just its tool:

```bash
openshell policy get metrics --full -o json
```

And tail an agent's **audit trail** — every egress decision it made, allowed or denied:

```bash
openshell logs metrics --source sandbox
```

When you're **done with the workshop** and want to remove the fleet, run this — *don't* run it now, it tears everything down:

```bash-ref
./scripts/fleet.sh down fleet.txt
```

The policy it writes per agent is the governance point — **specific to the tool**. The `metrics` agent gets *only* this beyond the registry:

```yaml
network_policies:
  in-cluster-registry: { … }                 # every agent: install skills from your registry
  tool-egress:
    name: tool-egress
    endpoints: [ { host: kps-kube-prometheus-stack-prometheus.monitoring.svc.cluster.local, port: 9090, access: full } ]
    binaries: [ { path: /usr/bin/node }, { path: /usr/bin/curl } ]
```

So `metrics` can reach Prometheus and nothing else; `logs` and `events` each get Loki:3100 and nothing else; `analyst` gets **no `tool-egress` at all**. Try to make one reach another's backend and the proxy denies it — visible on the [Live](/learn/live) **Policy decisions** tab.

## Each agent's soul — the role

The same `IDENTITY.md` / `SOUL.md` files from *Identity & Soul*, one set per specialist, are what make them behave differently — `fleet.sh` stages them from `manifests/openclaw/fleet-roles/<name>/`. Here's the essence of each (you can read the full files on the [Fleet](/learn/fleet-status) page, each with its on-disk path):

```markdown
# Scout 🔎 — logs/SOUL.md
Backend: Loki. Read the pods' OWN log output:  {namespace="demo"}
→ which pods are logging, the error lines, and which new pods are silent (never started).

# Gauge 📈 — metrics/SOUL.md
Backend: Prometheus. Quantify the fault:  kube_pod_container_status_waiting_reason{namespace="demo"}
and kube_pod_container_info{...} → the waiting reason + healthy-vs-failing image. Numbers, not narratives.

# Trace 🧵 — traces/SOUL.md
Backend: Tempo. Search recent traces of the service → is the new version serving any traffic at all?

# Probe 🛎️ — events/SOUL.md
Backend: Loki — but a DIFFERENT stream than Scout: the Kubernetes event feed an event-exporter ships in,
  {job="kubernetes-event-exporter"} |= "demo"  → the control plane's own words ("Failed to pull image …").

# Sage 🧠 — analyst/SOUL.md
NO egress, NO tools. Reason only over the others' findings → conclude the root cause + recommend the image.
```

So **`logs` and `events` share the Loki backend but read different things**: Scout reads the
*application's* log lines (`{namespace="demo"}`), while Probe reads the *cluster's* event stream
(`{job="kubernetes-event-exporter"}`) — the kubelet/scheduler's own account of what went wrong.
**Identity + soul + a tool-scoped policy = a specialist** — and you can read every line of what shapes it.

## The `cluster-telemetry` skill — how a sealed agent reads its backend

There's a catch the SOULs above gloss over: OpenClaw's built-in `web_fetch` tool **refuses
private/internal/cluster addresses** (an SSRF guard). Loki, Prometheus and Tempo all live on
`*.svc.cluster.local` (private ClusterIPs) — so a sealed agent *can't* use `web_fetch` to read
them. That's what the **`cluster-telemetry` skill** is for. `fleet.sh` stages it into every agent
that has a backend, at `/sandbox/.agents/skills/cluster-telemetry/`:

- **`SKILL.md`** — tells the agent: your backend is in-cluster, so don't use `web_fetch`; use this
  skill via the `exec` tool, run ONE query, report the concrete values.
- **`tq.js`** — a tiny script the agent runs as `node …/tq.js '<url>'`. It shells out to **`curl`**
  (an egress-allowed binary), so the call is still bounded by the agent's single-backend policy —
  **nothing is weakened, the SSRF guard stays on everywhere else.** It then prints a *compact*
  summary (a few lines), which matters: dumping raw telemetry JSON back into a small model
  overflows its context. It understands Prometheus vectors, Loki streams, Tempo searches, and
  renders Kubernetes events readably (`[Warning] Failed: Failed to pull image … (Pod/shop-…)`).

So every agent reaches its backend the **same governed way** — `curl` under its egress policy —
and you can read the whole skill (it's ~40 lines) in `manifests/openclaw/skills/cluster-telemetry/`.

<Callout type="note" title="Two kinds of 'skill' — don't confuse them">
The <strong>cluster-telemetry</strong> skill above is staged into every agent automatically (it's
<em>how</em> they read a backend). The <code>skill</code> column in <code>fleet.txt</code> is a
<em>separate, optional</em> hook: a first-class tool installed from your private registry (Part V),
e.g. <code>@workshop/promql</code> giving a typed <code>promql(query)</code> tool. It's <code>-</code>
by default — the agents work fine with just cluster-telemetry + their SOUL.
</Callout>

<Callout type="note" title="Why this stays safe to scale">
Adding a fifth agent is one more line in <code>fleet.txt</code> + one persona folder. It never widens the trust boundary: the new agent reaches exactly one more host, that grant is in its policy, and every call it makes is in the audit log.
</Callout>

Bring the fleet up, confirm `status` shows them Ready, then head to the next chapter to drive them.
